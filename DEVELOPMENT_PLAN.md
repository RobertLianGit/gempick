# 「心跳之间」核心开发文档

最后更新：2026-07-19

状态：MVP 已实现，可本地运行并静态构建。本文描述当前代码行为，不再是早期提案。

## 1. 产品定位

产品名：**心跳之间**

SEO 标题：**心跳之间｜邓紫棋歌曲个人 Pick 体验**

首页核心文案：

- 当 G.E.M. 邓紫棋遇上世界杯；
- 先进入预选阶段：十首种子歌自动入围，另外六个席位由你决定。组成歌曲 16 强后进入 Pick 阶段，每次两首相遇、Pick 一首，一路走到决赛；
- 开始我的 Pick。

这是一段借用世界杯节奏的个人 Pick 旅程，不是客观评选、公开投票或真实体育赛事。第一步是预选阶段：10 首种子歌自动入围，用户从全部作品中选择另外 6 首，组成歌曲 16 强。第二步是 Pick 阶段：16 强随机两两相遇，每次 Pick 一首进入下一轮，依次经过 8 强、4 强和决赛。完成 15 次 Pick 后，得到“我最 Pick 的 G.E.M. 邓紫棋的歌”、覆盖 16 首歌的个人排行榜和完整 Pick 过程。

产品价值包括两部分：

1. 帮助用户得到一个只属于当下的个人结果；
2. 在 Pick 过程中重新认识每首歌的主题、发行背景和版本信息，并鼓励用户在不熟悉时去正规音乐平台试听。

## 2. 项目边界

### 当前必须保持

- 非官方、非商业粉丝项目；
- 与艺人本人、工作室、唱片公司和相关团队不存在隶属或授权关系；
- 不托管音频，不展示歌词、MV、专辑封面、艺人照片、演唱会照片或官方 Logo；
- 不建立账号，不收集姓名、邮箱、手机号或精确位置；
- 不生成全站排名，不使用“最强歌曲”“第一名”“正确答案”等权威暗示；
- 作品事实和编辑摘要保留来源链接；
- 外部试听只跳转至正规音乐平台，不嵌入或抓取播放器。

### 当前不做

- 登录、注册、用户资料和云端同步；
- 评论、投稿、图片上传和社交互动；
- 广告、打赏、会员、票务、周边和赞助；
- 用户总体排行榜；
- 复杂后台和公开结果链接；
- 未取得授权的封面、音频、歌词或肖像素材。

`.xlsx` 只作为 Numbers 的导出副本，不作为独立修改源。页面实现应优先读取和遵循 Numbers 中“建议文案”列。2026-07-19 用户确认表内修改后的文案均视为已确认。

| 内部概念 | 用户界面文案 |

## 4. 当前用户流程

### `/` 首页

首页负责让用户在首屏理解产品，并在开始前看到最终结果会是什么样：

- 品牌“心跳之间”；
- 确认版核心文案与“开始我的 Pick”按钮；
- 世界杯概念说明：预选阶段 → 歌曲 16 强 → 8 强 → 4 强 → 决赛 Pick；
- 15 张发行、129 个专辑录音版本，并纳入精选、影视、公益与合作单曲索引；
- 非官方声明；
- “最后会得到什么”结果预览卡；
- 结果预览显示“我最 Pick 的 G.E.M. 邓紫棋的歌”、16 首歌曲、15 次 Pick 和旅程节点，并明确标注为非真实结果。
- 首页进入选歌和说明页的入口使用带尾斜杠的原生相对链接，确保 Safari 在 GitHub Pages 项目子路径下也能直接跳转；装饰轨道不得拦截点击。

### `/select` 预选阶段

页面标题：**十首种子歌，六个席位由你决定。**

行为：

- 首次进入自动入围固定的 10 首种子歌；
- 种子歌为：`回忆的沙漏`、`A.I.N.Y. 爱你`、`泡沫`、`多远都要在一起`、`来自天堂的魔鬼`、`光年之外`、`倒数`、`句号`、`新的心跳`、`超能力`；
- 种子歌不占 6 个自选席位，也不能从预选阵容中移出；
- 用户从全部作品中逐首选择 6 首，和种子歌一起组成固定的歌曲 16 强；
- 专辑卡显示年份、发行类型、歌曲数量和抽象轨道，展开后逐首操作；
- 为避免一次加入超过六个席位，不再提供整张专辑加入；
- 预选页面使用吸顶进度栏，始终显示“已选 N / 16 首”、可视化进度、六个自选席位完成状态和自动保存状态；
- 状态栏滚动时固定在浏览器顶部，手机与桌面都不遮挡主要内容；
- 选满 6 个自选席位后，同一吸顶栏内的“进入 Pick”按钮立即解锁，用户不需要回到页面顶部；
- 顶部提供“重新开始 Pick”，确认后清除自选歌曲、相遇顺序和 Pick 进度，并回到十首种子歌。

### `/wildcards` 逐首选择六个席位的兼容入口

主流程已经把六个自选席位并入 `/select`；该页面只保留给旧书签或直接访问的用户。

行为：

- 顶部显示当前数量、剩余数量、自动保存状态和“开始相遇”按钮；
- 顶部同样提供“重新开始 Pick”，确认后返回候选歌曲页；
- 按钮位于候选内容上方；
- 候选歌曲是紧凑方格矩阵，而不是两列长列表；
- 桌面每行 8 个方格，平板 6 个，手机 3 个；
- 每个方格显示歌曲名、专辑、年份和“加入候选/移出候选”；
- 支持按歌曲名和专辑名搜索；
- 达到 16 首前“开始相遇”不可用。

### `/bracket` 歌曲 16 强对阵

页面标题：**你的歌曲 16 强，准备相遇。**

显示：

- 10 首种子歌；
- 6 首用户选择；
- 16 强规模；
- 15 次 Pick；
- “生成 16 强对阵”按钮；
- `GEM-{seed}` 格式的旅程编号；
- 第一段相遇预览和“开始 Pick”按钮。

生成结果后必须稳定。刷新只读取本机已保存结果，不重新洗牌。正式 Pick 阶段固定使用 16 首，因此不会出现附加预选或稍后登场。

### `/play` Pick 阶段

核心标题：**这一轮，你 Pick 哪一首？**

每一边必须保持完全对称，并同时展示：

- 歌曲名、专辑和发行年份；
- 自行整理的“作品线索”；
- 可展开的专辑发行背景；
- 语言、唱片信息和资料来源；
- “Pick 这首”按钮；
- 明显的“去音乐平台听听”入口。

试听引导：

- 页面顶部提示“先去听听，再回来 Pick”；
- 第一次点击任意试听入口时询问常用平台，可选 QQ 音乐、Apple Music、网易云音乐和酷狗音乐；
- 平台偏好只保存在当前浏览器，之后 Pick 页、反馈页和结果页共同沿用；
- 全站页脚允许随时更改平台；
- 外链在新标签页打开；
- 选择 Apple Music 时，如果发行来源中有精确发行页则优先打开，否则进入歌曲搜索页；
- 其他平台进入包含“邓紫棋 + 歌名”的官方搜索页；
- 明确告诉用户打开外链不会丢失进度；
- 不嵌入播放器，不自动播放。

Pick 后显示：

- “你 Pick 了这一首，它将进入下一轮”；
- “另一首歌这一次没有被 Pick”；
- 歌曲主题摘要；
- 语言和发行背景；
- 继续 Pick 或撤回刚才的 Pick。

世界杯式轮次固定显示：`16 强 Pick` → `8 强 Pick` → `4 强 Pick` → `决赛 Pick`。

辅助操作：

- 左右方向键快速 Pick 两边歌曲；
- 撤销最近一次 Pick；
- 暂停并看看完整旅程；
- 进度文案显示已完成次数和总次数。

### `/result` 结果页

核心文案：

- 我最 Pick 的 G.E.M. 邓紫棋的歌；
- 最终 Pick 的，是这一首；
- 这不是哪首歌“最好”，只是这一刻，它最对你的心跳。

结果包含：

- 最 Pick 的歌曲、专辑和年份；
- 最初歌曲数量；
- 总 Pick 次数；
- 最 Pick 歌曲被 Pick 的次数；
- 最后一次遇见；
- “它走过的路”；
- 可展开的完整旅程；
- “去听这首歌”，直接使用已选择的听歌平台；
- “生成结果图片”，在浏览器内生成 4:3 PNG 并显示保存预览；
- “重新开始 Pick”。
- 完整个人排行榜：按歌曲最后一次参与 Pick 的阶段形成第 1 名、第 2 名、第 3–4 名、第 5–8 名等并列层级，并覆盖全部候选歌曲。

排行榜不做伪精确排序。同一阶段没有发生彼此比较的歌曲必须显示为并列；如果未来需要严格的 1–N 名次，必须另加排序比较流程，不能从当前单败选择中虚构。

结果图片格式：

```text
我最 Pick 的 G.E.M. 邓紫棋的歌
{歌曲名}
{专辑名} · {年份}
最终 Pick 的，是这一首。
{歌曲 16 强 / Pick 次数 / 被 Pick 次数}
心跳之间｜非官方粉丝项目
```

结果分享不得依赖 Clipboard API。局域网 HTTP 地址在 Safari 中不属于安全上下文，剪贴板写入会失败；PNG 必须通过 Canvas 在本机生成。

### `/about`、`/legal` 与错误页

- `/about` 解释预选阶段、Pick 阶段、自动保存、作品背景和试听入口；
- `/legal` 标题为“关于这个非官方发起的粉丝项目”，法律正文保持正式清楚；
- `app/error.tsx` 使用“这片星轨暂时没有连上”和“再试一次”；

## 5. 内容与运行时数据模型

源数据：`data/gem-discography.json`。专辑录音版本与 `standalone_and_featured_releases` 中的精选、影视、公益和合作单曲都会进入候选池。

运行时目录转换在 `src/data/catalog.ts` 完成。

```ts
type Track = {
  id: string;                 // release-id--position 或 single--year--index
  title: string;
  titleEn?: string;
  albumId: string;
  albumTitle: string;
  releaseYear: number;
  language: string[];
  summary: string;            // 自行整理的作品线索
  albumIntroduction: string;  // 专辑发行背景
  label?: string;
  version?: string;
  sourceUrls: string[];
};
```

不同语言、现场、混音和重录版本使用不同 ID。创建旅程前只按录音版本 ID 去重，不按歌曲标题去重。

## 6. 相遇算法

当前产品固定用 16 首歌进入 Pick 阶段，因此用户实际经历始终是 15 次 Pick：16 强 8 次、8 强 4 次、4 强 2 次、决赛 1 次。底层仍保留 16–128 首通用单败树能力，作为算法边界测试和未来扩展，不暴露在当前界面中。

令 `N` 为去重后的歌曲数量，`S` 为不大于 `N` 的最大 2 的幂：

```text
N = 16  → S = 16，附加 Pick 0 次
N = 17  → S = 16，附加 Pick 1 次
N = 32  → S = 32，附加 Pick 0 次
N = 43  → S = 32，附加 Pick 11 次
N = 64  → S = 64，附加 Pick 0 次
N = 100 → S = 64，附加 Pick 36 次
N = 128 → S = 128，附加 Pick 0 次
```

附加 Pick 数量为 `N - S`。附加 Pick 中被 Pick 的歌曲与稍后登场的歌曲共同组成 `S` 首正式顺序。总 Pick 次数始终为 `N - 1`。

算法要求：

1. 用随机种子初始化可复现伪随机数；
2. 使用 Fisher–Yates 洗牌；
3. 同一 `seed + selectedTrackIds` 必须生成同一结构；
4. 生成后保存完整结构，不在恢复时重新计算；
5. 通用算法中的附加预选位置和稍后登场位置都由同一个种子决定；
6. 不设置热门保护或同专辑回避；
7. 16–128 首必须无重复、无遗漏并且可以结束。

## 7. 状态与本机保存

localStorage key：

```text
gempick:lastsong:v1
```

听歌平台偏好使用独立 key：

```text
gempick:music-platform:v1
```

虽然产品品牌已经更名，MVP 暂时保留原 key，避免升级时丢失已有本机进度。只有明确设计迁移逻辑后才能更改。

保存：

- 已选歌曲 ID；
- 随机种子和旅程编号；
- 全部相遇节点；
- 每次 Pick；
- 当前进度和更新时间。

不保存：

- 姓名、邮箱、手机号；
- 精确位置；
- 云端个人结果；
- 未经同意的分析数据。

重新开始必须显示：

> 重新开始后，之前的 Pick 记录会被清除。确定要从头来吗？

## 8. 技术架构

- Next.js 16 App Router；
- React 19；
- TypeScript 严格模式；
- 自定义全局 CSS，不使用 Tailwind；

## 10. MVP 验收标准

新用户应当能够：

1. 在首页理解这是个人 Pick 旅程，并看到最终结果示例；
2. 明确知道项目非官方；
3. 在预选阶段看到固定自动入围的 10 首种子歌；
4. 从全部作品中选择另外 6 首，组成歌曲 16 强；
5. 在内容上方随时看到六个自选席位的完成状态、保存状态和下一步按钮；
6. 生成稳定不变的 16 强对阵；
7. 在每次 Pick 前理解两首歌的作品线索和发行背景；
8. 不熟悉时前往音乐平台试听并安全返回；
9. 刷新后继续刚才的 Pick；
10. 撤销最近一次 Pick；
11. 得到唯一的“我最 Pick 的 G.E.M. 邓紫棋的歌”；
12. 直接前往自己选择的音乐平台试听结果歌曲；
13. 查看它走过的路和完整旅程；
14. 查看覆盖 16 强全部歌曲、同阶段明确并列的个人排行榜；
15. 生成不含封面、歌词、Logo 或官方排名暗示的结果图片；
16. 在手机上完成平台选择、试听跳转、排行榜浏览、结果图片预览与保存。
